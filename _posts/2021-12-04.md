# Centos 9 Streams
As I like to see the future of RHEL, I have installed Centos 9 Streams on one of my home servers. As expected, no third party repositories support Centos 9 yet. Not even EPEL. This is not a problem as I can run all the services as containers.

## Podman to the Rescue.
Podman is a daemonless, rootless, container engine. Podman can be run as root or non-root user. Podman is designed to be a drop in replacment for Docker. You can even  run Docker Compose files with a little extra setup.

Some other advantages of podman is the ability to generate systemd service files or kubernetes Pod yaml files. Podman can be great at converting your docker or docker-compose file to kubernetes yaml. 

## Deploying Jupyter Notebook with podman

Starting with something simple, we will deploy Jupyter Notebooks.

Podman will automatically create a volume for the image, but it is recommended that you specify a location or create a named volume.

Most guides recommend running something like:

```bash:
podman container run -d \
    --name jupyter \
    -p 6000:8888 \
    -v jupyter-work:/home/jovyan/work:Z \
    jupyter/scipy-notebook
```
But then we have to get the token from the terminal, running podman logs jupyter. Kind of clunky. 

What we can do is create our own token or disable tokens completely by adding "start-notebook.sj --NotebookApp.token='' after scipy-notebook. If you want to create a token add --NotebookApp.token='my_token', replacing my_token with your own token.

```bash:
podman container run -d \
    --name jupyter \
    -p 6000:8888 \
    -v /home/user/jupyter:/home/jovyan/work:Z \
    docker.io/jupyter/scipy-notebook start-notebook.sh --NotebookApp.token=''
```
With the above command we create a container named "jupyter", accessible from localhost:6000. Files will be saved in ~/jupyter. Using Jupyter’s scipy-notebook image from Docker.io. The notebook token is disabled.

If you are deploying on a system running SELinux, you need to add a z or Z to the end of volumes. If the volume will be accessed by more than one container, use z, otherwise use Z. If you want to create a container tailored SELinux policy, look into udica.  

We can check if jupyter was setup correctly by running:
```Dockerfile:
podman container ps
```
## Generating systemd service file

The other great feature of podman, we can generate a service file so systemd manages our containers for us.

First create the systemd user directory in your home directory.

```bash:
mkdir -p ~/.config/systemd/user
```

Next generate the systemd service file.

```bash:
podman generate systemd --user --new --name jupyter -f /home/user/.config/systemd/user/
``` 
This will create container-jupyter.service in ~/.config/systemd/user/.

We have to reload systemd
```
Systemctl --user daemon-reload
```

Stop the jupyter container
```bash:
podman stop jupyter
```
 Run:

```bash:
Systemctl --user enable --now container-jupyter.service
```
Check if everything worked
```
Systemctl --user status container-jupyter
```
